# Esbla Spark — File Metadata Standard v1

**Target repo path:** `docs/standards/current/esbla_file_metadata_standard_v1.md`
**Authority tier:** Standard (below doctrine, binding on all new or modified active files).

## 1. Identity lock

Every active file's metadata uses exactly: **Company: HAUTM. Platform: Esbla Spark. Author: Usman Hussain. Owner: Usman Hussain.** Prohibited in metadata, file headers, authorship blocks, ownership blocks, and attribution statements: "generated by," "assisted by," "AI-assisted," "Codex-authored," "ChatGPT-authored," "Claude-authored," "Genesis-authored," or any equivalent authorship attribution. This prohibition does not prevent doctrine from discussing AI tools, AI governance, AI review, AI drafting workflows, three-AI stress tests, or AI product capabilities where those are operational subjects rather than authorship attribution. AKTI, AKTI ERP, AKTI Spark, and Arfa-Karim-specific naming appear only in files with `status: historical`, under archive paths, or in active transition lines explicitly listed in `legacy_name_map_v5_0.json`.

## 2. Suite version vs file version

Two distinct fields, always both present:

- `suite_version` — the documentation-suite generation the file belongs to. Current: **v5.0**. Bumped only by a suite rebaseline ratification.
- `file_version` — the document's own version (e.g., blueprint `v3.1`, business logic `v2.0`). Bumped per document on substantive change.

A file_version bump never implies a suite bump; a suite bump does not renumber files. This separation is the permanent fix for the v4.1/v3 version-conflation problem.

## 3. Required fields and status vocabulary

Required: `company, platform, author, owner, suite_version, file_version, created, last_updated, status, document_type, scope` (one-line purpose). Status vocabulary (closed set): `draft | for_ratification | active | authoritative | historical | deprecated`. `authoritative` is reserved for ratified doctrine and standards; `active` for operating documents; `historical` for archived evidence (never edited, only moved); `deprecated` for superseded files awaiting archive.

## 4. Formats by file type

**Markdown:** YAML frontmatter exactly as this file demonstrates.

**TypeScript / TSX / JavaScript:** compact block comment header, only where useful:

```ts
/**
 * Company: HAUTM | Platform: Esbla Spark
 * Author: Usman Hussain | Owner: Usman Hussain
 * Suite: v5.0 | File: v1.0 | Created: 2026-06-13 | Status: active
 * Scope: One-line purpose.
 */
```

**JSON:** a `$metadata` object with the same fields, only where the consuming schema tolerates unknown keys. If it does not, metadata lives in the registry (Section 5), never in the file.

## 5. Unsafe-file rule and the metadata registry

Never add headers or metadata fields to: lockfiles, binaries, generated build artifacts, strict generated outputs, strict-schema JSON, third-party vendored files, minified files, snapshots whose expected output would change, `.env` files, and migrations (unless a migration-specific approval exists). These files are recorded externally in `docs/process/registries/file_metadata_registry_v5_0.json`. A parser, schema, or snapshot failure caused by metadata is a defect: revert the edit and move that file class to registry-only (blueprint risk row 26).

## 6. Single-source-of-numbers rule

All governed numeric thresholds live **only** in Blueprint v3.1 Appendix A (the Parameter Registry). Every other suite document references parameters by ID (`P-ID`, up to the current maximum defined in Blueprint Appendix A) and never restates the value. A restated number anywhere outside Appendix A is a doctrine defect: fix by replacing with the P-ID. This prevents table-vs-prose drift across the suite.

## 7. Naming, paths, and legacy references

- All new doc and process paths use `lower_snake_case`; validated by `docs:check-lower-snake-case-paths`.
- Legacy-name policy: `docs:check-legacy-names` scans active paths for `AKTI|akti|AKTI Spark|Arfa Karim`; any hit on an active file fails CI unless the exact file/line/term is listed in `docs/process/registries/legacy_name_map_v5_0.json`. Historical/archive paths are exempt. Broad path allowlisting is prohibited except for archive roots.
  The legacy-name map may contain legacy terms because it is the exception registry; validators must not count the registry itself as an unclassified legacy-name violation.
- Filenames carry their own file_version suffix (`_v3_1`, `_v2`); the suite version is carried in frontmatter only, never in filenames (founder ruling — see suite manifest declined-items log).

## 8. Adoption

Applies to every file created or modified from Refactor Wave 1 onward. Pre-existing files gain metadata when first touched or during Wave 4 renames; mass retro-fitting of untouched historical files is prohibited.

End of standard v1.
